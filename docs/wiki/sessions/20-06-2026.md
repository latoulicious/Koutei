# Summary

Added stamina rest/recovery — the rule that makes the timeline non-trivial.
Unassigned operators now recover instead of sitting idle, so a drained operator
rotates back into production a later slice.

# Files Touched

- `internal/domain/operator.go` — `Operator` gains `StaminaMax`, `Regen`.
- `internal/domain/rules.go` — new `RecoverStamina(stamina, regen, max)`.
- `internal/domain/rules_test.go` — `TestRecoverStamina` (cap, no-op, never-lowers).
- `internal/solver/solver.go` — unassigned operators recover after the assign loop.
- `internal/solver/solver_test.go` — `TestSolve_RestRecovery` (4-slice rotation).
- `docs/wiki/modules/domain.md`, `modules/solver.md` — rest moved out of deferred.

# Previous Behavior

Drain-only: unassigned operators sat idle; over a horizon all hit 0 stamina and
output died, with no rotation to optimize.

# New Behavior

`RecoverStamina` = `min(stamina+regen, max)`, never lowering stamina (zero max =
no-op). Solver rests every unassigned operator each slice. Two operators sharing
one slot now alternate as each drains and recovers.

# Reason For Change

Make multi-slice scheduling meaningful before Phase 2 branch-and-bound — B&B on a
drain-only model would optimize an incomplete world.

# Risks

Low. New `Operator` fields default 0 → recovery is a no-op for existing callers
and tests (behavior preserved). Pure, deterministic, fully tested.

# Notes

- Remaining deferred trims: `moodBonus` aura, `Level`, `target_priority` matching.
- gofmt clean · `go vet ./...` clean · `go test ./...` green.

# Summary

Phase 2: branch-and-bound solver. Greedy is per-slice myopic — it can burn a
strong operator now that a later slice needed. `SolveBranchBound` maximizes
cumulative efficiency over the horizon instead, seeded by the greedy `Solve` as
its incumbent lower bound. New exported function; greedy stays untouched.

# Files Touched

- `internal/solver/branch_and_bound.go` (new) — `SolveBranchBound` + DFS search,
  `maxSliceEfficiency` bound, dominance frontier, candidate enumeration.
- `internal/solver/branch_and_bound_test.go` (new) — beats-greedy (3.9 vs 3.8),
  never-worse-than-greedy, determinism, slot capacity, zero-stamina prune.
- `docs/wiki/modules/solver.md` — B&B section added; moved out of Deferred.

# Previous Behavior

Only the greedy `Solve` existed: locally optimal each slice, globally suboptimal.

# New Behavior

`SolveBranchBound(ops, stations, horizon)` returns the cumulative-efficiency-
maximizing schedule via DFS branch-and-bound with three prunes: greedy incumbent,
admissible per-slice upper bound, and dominance (≥ acc and ≥ stamina everywhere).
Proven optimal under a deterministic `maxNodes` budget; best-found (≥ greedy)
otherwise. Deterministic and pure like `Solve`.

# Reason For Change

PLAN §5 / solver.md Phase 2: B&B with dominated-branch pruning replaces the
per-slice greedy choice.

# Risks

Low-moderate. New function — `Solve` and its tests untouched, output types and
domain rules unchanged. Ceiling: per-slice assignment space is exponential in
roster size; the node budget bounds worst-case and degrades to best-found (never
below greedy). Documented in solver.md.

# Notes

- Greedy `Solve` kept as the incumbent seed (standard B&B practice), not deleted.
- `availableSorted` duplicates `solveSlice`'s filter intentionally — keeps the
  greedy path untouched rather than refactoring shared code into the feature diff.
- Still deferred: `moodBonus` aura, `Level`, `target_priority` matching.
- gofmt clean · `go vet ./...` clean · `go test ./...` green · `-race` clean.
