# Summary

Phase 4: the TypeScript SPA. A local-first, dependency-light interface (Vite +
vanilla TS + Tailwind v4) under `web/` that bundles `tools/seed/seed.json`, lets
the user build a roster + station matrix, resolves the seed to the backend's
numeric payload client-side, posts to `POST /api/v1/optimize`, and renders the
returned timeline as a Gantt. First frontend code; the Go backend is untouched.

# Files Touched

- `web/package.json`, `tsconfig.json`, `vite.config.ts`, `index.html` (new) —
  scaffold; dev proxy `/api` → `:8080`, `fs.allow ['..']` to bundle the repo seed.
- `web/src/style.css` (new) — Tailwind v4 `@theme` Control-Room palette (PLAN §4).
- `web/src/seed.ts`, `state.ts`, `payload.ts`, `api.ts`, `context.ts`, `dom.ts`
  (new) — seed import + types, localStorage state, the seed→numeric mapping, fetch
  client, action surface, `el()` builder.
- `web/src/ui/roster.ts`, `stations.ts`, `timeline.ts`, `widgets.ts` (new) — the
  three columns + shared widgets.
- `web/src/payload.test.ts` (new) — vitest: mapping + array-order stability.
- `docs/wiki/modules/web.md` (new) — SPA contract.

# Previous Behavior

Backend-only: the engine answered `POST /api/v1/optimize` but nothing drove it; no
UI, the bundled seed was unused.

# New Behavior

A three-column SPA (Roster slate → Station matrix → Gantt timeline). Operators are
picked from the seed; a factory-skill line + level resolves to `skill_bonus`, an
optional mood line to `mood_bonus`; stamina via slider with a blocky bar.
`buildPayload` emits the numeric request in `state` array order; the timeline maps
response indices back through the same arrays (`↻` marks rotation swaps; a badge
shows total efficiency + `Executed in N ms`). Inputs persist to `localStorage`;
the solve result does not.

# Reason For Change

PLAN §149 Phase 4 — wrap the numeric engine in the interface; close the
end-to-end loop now that the seed exists.

# Risks

Low. New `web/` directory only; backend, solver, API contract, and seed file
unchanged. The mapping defaults are the one soft spot — stamina mechanics
(`drain_base`/`regen`/`stamina_max`) are placeholder game constants and mood-skill
classification is user-driven, both flagged rather than fabricated.

# Summary (entry 2 — skill-line default)

Small follow-up to the SPA: a new roster operator now defaults its skill line to
`manufacture_efficiency` instead of array-position-0, via `primarySkillLine` in
`payload.ts`. Same-day, frontend-only, no contract change.

# Files Touched

- `web/src/payload.ts` — new `primarySkillLine(op)`: manufacture line, else first,
  else 1.
- `web/src/main.ts` — `addOperator` uses it for the `skillLine` default.
- `web/src/payload.test.ts` — covers prefer-manufacture / first-line / undefined.
- `docs/wiki/modules/web.md` — mapping note.

# New Behavior

Adding an operator pre-selects its production skill (`manufacture_efficiency`), so
the initial `skill_bonus` is meaningful rather than arbitrary array order. 3 of 26
seed operators change (manufacture skill wasn't first); 8 unchanged; 15 without a
manufacture skill fall back to the first line. User can still override.

# Reason For Change

Option (b1): wire the one sourced, unambiguous seed→domain mapping. Investigated
the deeper "real stamina/mood constants" path (b2) first and confirmed it blocked —
the daydreamer CDN archive is raw encrypted Unity VFS chunks, and neither
endfieldtools.dev's localdb (operators + manufacturing recipes only) nor the
community planners (factoriolab et al.) expose operator mood/drain/regen. Those
constants live only in the encrypted client; a full asset datamine is out of scope.
Stamina mechanics stay placeholder constants.
